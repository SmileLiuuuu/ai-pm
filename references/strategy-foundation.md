# 01_STRATEGY — 立项与战略记忆

本文件夹的定位：**立项与价值决策的依据信息**的存放地。目的是把「为什么做这版、做给谁、解决什么、不做什么」盘点清楚、记录清楚，让项目推进一段时间之后再次回顾时，仍能看到当时的判断依据。

---

## 文件夹内应维护的内容

### 1. 立项文档（价值与范围）

在 `01_STRATEGY/` 内维护一份**立项类文档**，聚焦**项目的价值决策依据**。命名可参考 [prd-protocols](prd-protocols.md) 中的 Strategy doc 约定（如 `v1_0_value_and_scope_YYYYMMDD.md`），具体文件名以项目与用户约定为准。

**文档目的：**
- 沉淀价值锚点、范围边界、目标用户与场景、痛点与成功指标、关键结构性决策；
- 作为「为什么做这一版 MVP」的单一事实来源，供 PRD 引用或简述，也供日后回顾。

**建议涵盖的维度（不强制结构，以与用户沟通中明确的信息为准）：**

| 维度 | 说明 |
|------|------|
| 项目基本信息 | 项目 ID、一句话描述（本版 MVP 的定位） |
| 价值锚点 | 核心价值、Why this problem / Why now；可用「当前共识 / 待确认」方式列出 |
| 范围边界 | In scope（首版聚焦）、Out of scope / 后续迭代 |
| 目标用户与典型场景 | 目标用户画像、2～3 个典型使用场景（可基于真实故事） |
| 不可忍的痛点与成功指标 | 用户原话级别的痛点；Tangible Anchors（成功时客观上会发生什么变化） |
| 关键 use case / 结构性决策 | 影响多模块的重大决策（如规划粒度、核心流程假设），便于后续 PRD 一致引用 |
| 下一步 / 版本假设 | 当前已澄清什么、未澄清什么；下一阶段要回答的问题 |

**与 PRD 的关系：** Framework PRD 的 Background & value anchors、MVP scope 可**引用或简要复述**本立项文档，不必在 PRD 中重复展开；价值与范围的一手依据以 01_STRATEGY 的立项文档为准。

**结构说明：** 上述维度为参考思路，**不强制**章节顺序或必选节。具体写什么、怎么写，以对话中与用户明确的内容为准。

---

### 2. DECISIONS.md（业务决策记录）

记录具体业务判断的「结论与理由」，便于团队查阅「当时为什么这么决定」。写法与触发时机见 [memory-system](memory-system.md) 中对 `docs/01_STRATEGY/DECISIONS.md` 的说明。

**与立项文档的关系：** 重大价值或范围相关的判断，既可在立项文档的相关位置保留（如引用或小结），也应在 DECISIONS 中记一笔，便于按决策 ID 追溯；两者互补，不替代。

---

## 使用习惯建议

- **何时创建/更新立项文档：** Scene 1（Value Discovery）或立项阶段，随价值、范围、用户与场景的澄清逐步写入或更新；可先有框架再补细节。
- **回顾时：** 做阶段复盘或启动新版本时，优先从 01_STRATEGY 的立项文档与 DECISIONS 回顾「当时的决策依据」，再决定是否调整范围或方向。
