# 03_DESIGN — 设计与中间产物（PRD 之后、开发之前）

本文件夹的定位：**PRD 定稿之后、真正开始开发之前**的设计与中间产物的统一存放地。PRD 回答「做什么」，本文件夹回答「怎么做、用什么话术、什么技术方案、界面长什么样」，开发再据此实现。

---

## 设计阶段与产出顺序（建议）

1. **设计指导文档（文字版）** → 页面拆分、每页功能/内容/字段、设计原则；可含待定项，解决后整理入文档与 DECISIONS。
2. **关键设计原则**（粗粒度）与 **核心用户流程 + 线稿** 可并行推进。
3. **关键组件** 从流程中提取 → AI 生成 **组件 demo** → 人对 demo 提修改建议。
4. **定稿视觉与 handoff** → 高保真、design tokens、每屏交付说明。

以下按文件夹说明各产出的定义与约定。

---

## ui_ux/ — 设计指导文档与设计原则

### 设计指导文档（页面拆分）

- **是什么：** PRD 全部输出之后，由 AI 生成的一份「页面拆分」文档，详细描述每个页面/视图需要包含哪些功能、哪些内容，具体到字段粒度。相当于产品经理向设计师讲完需求后，设计师脑中形成的、用于指导后续细节设计的**框架**。
- **信息密度：** 框架阶段不必一次写满所有细节；在 AI 协作下可先用「先框架、再逐项补充」的方式（如 plan 式写作），逐步得到相对完整的设计文档。
- **典型内容：** 全局设计原则、全局导航与信息架构、全局共享组件说明、各页面/视图的「内容区块」「视觉状态」「交互」「导航」等，以及上游 PRD 索引。可内含核心用户流程的线稿（ASCII/ Mermaid 或链接到 `prototypes/`）。
- **存放：** 此类文档放在 `ui_ux/`，例如 `ui_ux/design_handoff_v1_0_YYYYMMDD.md`。

### 设计原则

- **是什么：** 相对粗粒度的关键设计原则，贯穿所有页面，在开始具体页面前应达成一致。
- **存放：** 可单独文件（如 `ui_ux/design_principles.md`），或合并在设计指导文档的靠前章节。

### 待定项标记与整理策略

设计指导文档撰写过程中，必然会出现不明确之处。约定如下。

**1. 标记**

- 在文档中用统一格式标记需再次确认或需人类补充细节的问题，例如：`[待定项-001]` 或 `[待定-001]`。
- 建议在文档末尾设「待定项汇总」表，列出编号、问题简述、涉及页面/组件、关联 PRD 或 TODO。

**2. 反馈方式**

- 人对每个待定项可：在文档内直接回复（如 `[Answer to 待定项-001]` + 结论），或通过对话与 AI 讨论后再把结论写回文档。

**3. 待定项确认后的整理（与批注整理策略类似）**

- **结论明确化：** 在文档相关位置保留结论，用引用/Answer 格式（如 `[Answer to 待定项-001]`），不删除原待定项标记，便于追溯。
- **关键业务判断留存：** 重要判断除写在文档内外，如确有必要，同步写入项目的 DECISIONS（如 `docs/01_STRATEGY/DECISIONS.md`）。
- **与 PRD 的一致性：** 若结论与原有 PRD 设计冲突，讨论时就应指出；若最终决定与当初 PRD 不一致，**必须对 PRD 进行修改**。修改前需与用户确认。修改方式二选一：
  - **方式 A：** 原文直接覆盖。
  - **方式 B：** 原文用划线格式删除旧版，再在文档中补充新设计。
- **衍生 TODO：** 待定项可能衍生出后续任务，这些 TODO 需记录到项目 TODO（如 `docs/TODO.md` 或 `.ai_pm/TODO_STACK.json`），并在待定项汇总中注明对应 TODO 编号。

整理完成后，即得到一份可指导线稿与细节设计的**文字版设计指导方案**。

## prototypes/ — 线稿与组件 demo

- **线稿：** 核心用户流程上的页面，以线稿形式表达（流程图、页面线框图等）。格式不限定：Figma 链接、导出图、Markdown/ASCII 图、Mermaid 等，以能支撑评审和开发理解为准。可与设计指导文档同仓库，或在文档中引用 `prototypes/` 内文件。
- **关键组件 demo：** 从核心业务流程中提取的关键组件，由 AI 生成可交互或可预览的 **demo**，便于人基于 demo 表达修改建议，再迭代设计。demo 文件可放在 `prototypes/`（如按组件分子目录或命名区分）。

---

## ai_prompts/ — 模拟的 AI Prompt

- **放什么：** 与产品功能相关的 AI 话术与 Prompt 草稿，用于在开发前对齐「AI 怎么说、怎么响应」。
- **示例：** 某功能的 system/user prompt 草稿、示例对话、边界 case 的预期话术、多轮对话脚本。

---

## tech_design/ — 技术设计文档

- **放什么：** 开发前或与开发协作产出的技术级设计。
- **示例：** 模块/服务划分、关键接口约定、数据流说明、技术选型与风险。格式以 Markdown、Mermaid 等可版本管理的文本为主。

---

## 使用习惯建议

- **命名：** 文件命名与 PRD 功能/模块或版本对应（如 `ui_ux/design_handoff_v1_0_20260227.md`、`ai_prompts/order_chat.md`）。
- **与决策记录：** 重要设计决策（含待定项结论、技术选型等）确认后，除在 03_DESIGN 中更新外，可在 GENEALOGY_LOG 或 DECISIONS 中记一笔，便于回溯。
